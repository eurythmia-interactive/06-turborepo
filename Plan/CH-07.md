# **Chapter 7: Quality Verification, CI/CD, & Boilerplate Checklists**

**Goal:** Establish automated testing systems using **Vitest 4** and transactional database rollbacks, build continuous integration pipelines using GitHub Actions with modern caching engines, implement configuration validations via **Zod 4**, and configure production deployment plans.

---

### **Task 7.1: Automated Integration Test Pipelines (Vitest 4 + Prisma 7)**

_Objective: Implement a fast, isolated integration test suite where backend routes execute query loops directly against a local test database, rolling back mutations automatically using transaction sandboxes._

- **Subtask 7.1.1: Vitest 4 Testing Workspace Setup**
  - Install **Vitest 4** as a development dependency within `apps/api` and `packages/database`.
  - Create a dedicated `vitest.config.ts` file in the backend workspace. Configure the configuration to target the Node runtime and define the tests pattern matching files (such as `*/*.integration.test.ts`).
  - Configure the projects parameter block inside the root testing config using Vitest 4’s projects array instead of legacy workspace definitions to organize independent package execution scopes.
- **Subtask 7.1.2: Dynamic Database Schema Isolation**
  - Create a global setup file at `apps/api/test/setup.ts` to run before test execution.
  - Configure the setup file to dynamically generate a unique database schema name (e.g., matching the current Vitest worker thread ID) for every parallel test worker to prevent concurrent tests from altering each other's data tables.
  - Configure the database client to apply Prisma 7's raw migration deployments (`prisma migrate deploy`) on these dynamically constructed schemas at worker initialization, tearing down the test schemas during the final worker cleanup stages.
- **Subtask 7.1.3: Prisma 7 Programmatic Sandbox Transactions**
  - Design a test database sandbox utility at `apps/api/test/db-sandbox.ts`.
  - Configure a wrapper helper that executes each individual test block inside an isolated Prisma `$transaction` block.
  - Ensure the helper automatically issues a transactional rollback command immediately after the test assertion block executes, keeping the test database schema clean and restoring original index values without executing slow database truncations or resets.
- **Subtask 7.1.4: Endpoint Request Mocking**
  - Initialize the **Supertest** package to handle mock HTTP request cycles against the NestJS backend container during integration runs.
  - Configure test assertions to evaluate returned JSON payloads, HTTP headers, and status codes (e.g., verifying that a login attempt rejects bad schemas with strict status codes and accurate Zod error lists).

### **Task 7.2: Continuous Integration Gates (GitHub Actions + pnpm 11)**

_Objective: Create a GitHub Actions workflow to run quality validation checks (type compilation, code analysis, lint rules, and test suites) on code pushes and pull requests._

- **Subtask 7.2.1: Master CI Workflow Blueprint**
  - Create the primary workflow file at `.github/workflows/ci.yml`.
  - Configure the pipeline triggers to run on every code push to the main branch and on every pull request target branch modification.
- **Subtask 7.2.2: Build Environment & Global Dependency Caching**
  - Configure the runner environment to use the latest stable Ubuntu machine image.
  - Instruct the action to set up Node.js pinning the environment strictly to **Node.js 24 LTS / 26 Current**.
  - Activate Corepack to initialize **pnpm 11**, leveraging pnpm’s native dependency store caching configuration keys to keep continuous integration boot sequences under 60 seconds.
- **Subtask 7.2.3: Turborepo Remote Cache Hookup**
  - Incorporate the official Turborepo GitHub Action to enable shared, secure remote compilation caching.
  - Map the pipeline tasks to Turborepo command execution keys, ensuring that compilation processes (`build`), linting checks (`lint`), and tests (`test`) skip execution for unmodified monorepo workspaces.
- **Subtask 7.2.4: Code Quality and Lint Validation Steps**
  - Add validation check gates inside the CI sequence:
    - **Type Checker:** Execute TypeScript compilation commands across all workspaces to catch typing drift early.
    - **Linter:** Execute Oxlint for rapid syntax checking, followed by full ESLint 10 checks to evaluate complex design rules.
    - **Formatter:** Execute Prettier check scripts to verify code styling conventions.
    - **Test Suite:** Spin up a local containerized database instance in the runner environment, run database migrations, and execute the Vitest 4 integration suite.

### **Task 7.3: Environment Schema Isolation (Zod 4 Environment Checking)**

_Objective: Build an environment configuration parser that verifies application parameters on bootstrap, protecting application servers from starting with broken or incomplete configurations._

- **Subtask 7.3.1: Global Configuration Contract Mapping**
  - Create an environment schema contract file at `packages/shared/src/env/env.schema.ts` using **Zod 4**.
  - Define properties for expected public client parameters (such as public API gateway URLs, support email directories, and active web widget keys) and private server credentials (such as database URLs, JWT encryption secrets, and cookie security flags).
  - Specify explicit default fallback parameters for development contexts, while forcing strict string properties without fallbacks in production configurations.
- **Subtask 7.3.2: NestJS Startup Ingestion and Validation**
  - Integrate the environment schema parser into NestJS’s primary configuration bootstrapping layer.
  - Configure the server bootstrap sequence inside `apps/api/src/main.ts` to execute Zod's strict `.parse()` method against `process.env` immediately upon execution.
  - If any configuration variables are missing or malformed, instruct the server to throw a descriptive configuration error detailing the mismatched keys and halt the server's bootstrap sequence early to prevent unstable runtimes.
- **Subtask 7.3.3: Next.js 16 Edge and Server Configurations**
  - Implement matching environment checking checks inside Next.js layout structures.
  - Differentiate public variables (such as parameters prefixed with `NEXT_PUBLIC_` meant for client browser exposure) from server-side secrets, ensuring that server-side credentials are never loaded or exposed to the client bundle.

### **Task 7.4: Deployment Configurations & Production Build Pipelines**

_Objective: Prepare the full-stack applications for deployment by building lightweight, secure, and performance-optimized application artifacts._

- **Subtask 7.4.1: NestJS Multi-Stage Docker Configurations**
  - Create a production-grade `Dockerfile` inside `apps/api`.
  - Implement a multi-stage compilation flow:
    - **Stage 1 (Builder):** Import monorepo dependencies, run pnpm installation steps, compile shared workspace contracts, and run NestJS build scripts to generate compiled Javascript distribution assets.
    - **Stage 2 (Runner):** Select a lightweight, secure Alpine base image, copy only production dependencies and compiled build assets, configure system variables, and configure the application container to execute as a non-privileged, non-root user.
- **Subtask 7.4.2: Next.js 16 Bundle Size Optimization**
  - Configure Next.js 16 build pipelines to verify bundle sizes and asset compilation.
  - Integrate a bundle analyzer plugin inside `next.config.js` to inspect component weights on build.
  - Ensure Tailwind CSS v4's modern CSS compiler outputs are optimized, minified, and mapped to modern, lightweight CSS variable files.
- **Subtask 7.4.3: Edge Caching and CDN Routing Strategies**
  - Formulate a production CDN strategy, separating dynamic App Router pages from static assets and public files.
  - Define explicit caching headers (e.g., `Cache-Control` rules) for public web widget assets, static page components, and API route responses to optimize response times for end users.
