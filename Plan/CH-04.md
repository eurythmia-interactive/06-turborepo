# **Chapter 4: NestJS Core Architecture & Request Lifecycle**

**Goal:** Establish a robust NestJS backend application utilizing **NestJS 11** on top of the newly standard **Express v5** engine, incorporating contract-first request validation pipelines and structured error handling patterns.

---

### **Task 4.1: Modular Service Hierarchy & Express v5 Bootstrapping**

_Objective: Scaffold the foundational NestJS application, configure its container dependency injection boundaries, and adapt routing logic to the Express v5 engine standards._

- **Subtask 4.1.1: Core Application Setup & Express v5 Adapters**
  - Navigate to the `apps/api` directory to initialize the backend workspace.
  - Install the stable **NestJS 11** core framework packages (`@nestjs/common`, `@nestjs/core`, `@nestjs/platform-express`).
  - Set up the server bootstrapper using NestJS’s integrated **Express v5** adapter. Note Express v5’s path route matching revisions, ensuring that route definitions avoid legacy wildcards and adopt named parameters (e.g., mapping wildcards to defined splat properties or custom bracket segments) to prevent routing mismatch errors.
- **Subtask 4.1.2: Root Module Configuration & Environment Isolation**
  - Create the master configuration wrapper `src/app.module.ts`.
  - Register a robust configuration provider that parses runtime environment setups utilizing a dedicated configuration class.
  - Incorporate a startup validation check that evaluates incoming environment variables against a predefined validation schema, throwing immediate bootstrap crashes if essential variables (such as relational connection strings, JSON Web Token secret keys, or CORS allowed domains) are missing.
- **Subtask 4.1.3: Database Module Wrapper Setup**
  - Construct the database client module wrapper at `src/database/database.module.ts`.
  - Configure the module to import and wrap the workspace-linked `@repo/database` package.
  - Establish a custom injection provider to wrap the global database client singleton, making sure to export this provider to make it inject-ready across separate feature packages.
- **Subtask 4.1.4: Domain-Driven Modules Directories**
  - Scaffold modular features paths within the application:
    - `src/user/`: Encapsulates user entities database interactions and profile retrieval pathways.
    - `src/auth/`: Contains authentication logic, credential processing, and secure session management.
    - `src/health/`: Implements a lightweight database pool, storage, and memory resource check.

### **Task 4.2: Shared-Contract Global Validation Pipe**

_Objective: Create a custom global NestJS validation pipe that parses, validates, and cleans client request payloads against Zod validation schemas before hitting controller methods._

- **Subtask 4.2.1: Custom Zod Validation Pipe Design**
  - Create the validation pipe class at `src/common/pipes/zod-validation.pipe.ts` implementing the core NestJS `ArgumentMetadata` lifecycle interface.
  - Equip the pipe’s constructor to accept specific target validation schemas, or dynamically read them from metadata bindings attached directly to controllers.
- **Subtask 4.2.2: Payload Parsing & Safe Schema Analysis**
  - Implement the pipe’s `transform` logic to evaluate request payloads against target Zod contract schemas.
  - Leverage **Zod 4**'s high-speed `.safeParse()` parser, avoiding expensive try-catch blocks and capturing syntax mismatch issues natively inside a structured analysis object.
- **Subtask 4.2.3: Nested Validation Errors Generation**
  - When validation fails, intercept the list of errors generated by the Zod parser.
  - Map these errors systematically to extract the target path location, specific failed constraints, and localized feedback messages.
  - Wrap these items into a specialized exception class (e.g., `ValidationException`) that inherits from NestJS's core `BadRequestException`, preparing the array for structured mapping in the exception filter.
- **Subtask 4.2.4: Global Pipe Registration**
  - Register the validation pipe inside `src/main.ts` as an application-wide execution gate using the global pipe mounting method.
  - Configure options to automatically strip away undeclared body properties to mitigate prototype pollution and parameter injection.

### **Task 4.3: Custom Standard Schema Interceptors**

_Objective: Establish routing decorators and interception gates to support uniform request verification schemas, future-proofing the application for multiple validation engines._

- **Subtask 4.3.1: Standard Schema Protocol Incorporation**
  - Design contract parsing adapters to comply with the standard validation protocol specifications (aligning with NestJS's roadmap for simplified validation toolchains).
  - Construct interface bridges allowing the server to seamlessly accept inputs from multiple schema validation tools without altering route handlers or controllers.
- **Subtask 4.3.2: Input Schema Binding Decorators**
  - Create custom request parameter decorators (e.g., `@UseSchema()`) at `src/common/decorators/use-schema.decorator.ts`.
  - Allow developer controllers to annotate route parameter fields (such as `@Body` or `@Query`) directly with a contract defined in `@repo/shared` to enforce compile-time verification checks.
- **Subtask 4.3.3: Output Serialization Interceptors**
  - Create a global serialization interceptor at `src/common/interceptors/serialize.interceptor.ts` to manage outgoing data structures.
  - Configure the interceptor to inspect outgoing data payloads, automatically stripping away restricted entity fields (such as security credentials, internal database reference states, or refresh tokens) based on schema exclusion directives.

### **Task 4.4: Centralized Error Mapping (Filters)**

_Objective: Construct a global NestJS exception filter to intercept database failures, validation issues, and server exceptions, formatting all errors into a unified API response._

- **Subtask 4.4.1: Centralized Exception Filter Implementation**
  - Construct the application-wide exception filter class at `src/common/filters/global-exception.filter.ts` decorated with the generic `@Catch()` parameter.
  - Implement logic to capture standard NestJS HTTP exceptions, database engine exceptions, and unhandled runtime engine crashes.
- **Subtask 4.4.2: Structured API Error Schema Formatting**
  - Format all error responses systematically. Ensure all outbound error payloads return a consistent, standard JSON structure containing:
    - `statusCode`: The numeric HTTP status code mapping.
    - `message`: A human-readable summary string describing the exception.
    - `errors`: An array or key-value dictionary containing fine-grained, target-field validation exceptions.
    - `timestamp`: The exact ISO date-time string when the exception occurred.
    - `path`: The relative URL target matching the failing route.
- **Subtask 4.4.3: Database Error Code Translators**
  - Implement specific translation mappings for core errors thrown by **Prisma 7** (e.g., parsing code records representing unique index violations, missing entity references, or foreign key constraints).
  - Instruct the filter to map database-level constraint exceptions into clean HTTP responses (like `409 Conflict` or `404 Not Found`) without revealing raw SQL, connection pools, or schema metadata to the client.
- **Subtask 4.4.4: Production-Safe Catch-All Branch**
  - Configure a final fallback block inside the filter to process raw, unhandled JavaScript `Error` exceptions.
  - Hide deep, descriptive runtime stack traces from production clients to mitigate information disclosure threats, returning a standard, safe status payload while saving the deep diagnostic trace locally.

### **Task 4.5: Global Request Logging & Auditing**

_Objective: Establish highly detailed request logging middleware to output performance metrics and trace correlation parameters._

- **Subtask 4.5.1: High-Performance Logging Middleware Configuration**
  - Configure global middleware at `src/common/middleware/request-logger.middleware.ts` implementing the NestJS `NestMiddleware` interface.
  - Intercept the request-response cycle to measure core transaction metrics: incoming HTTP request method, target request path, response size in bytes, and execution duration in milliseconds.
- **Subtask 4.5.2: Trace Correlation Identifiers**
  - Generate a unique transaction correlation identifier (such as standard UUID v4 strings) on every incoming HTTP request header.
  - Append this transaction correlation identifier to every single log entry generated throughout the execution cycle to allow developers to search log archives for the exact steps that occurred during a specific user transaction.
- **Subtask 4.5.3: Production-Grade JSON Log Outputs**
  - Configure the logger logic to automatically format and output logs as newline-delimited JSON strings when running in production configurations, making it compatible with log parsing engines like Datadog, Databricks, or Elasticsearch.
  - Utilize local, human-readable terminal log formatting options when executing the development pipeline on a local machine to simplify troubleshooting.
