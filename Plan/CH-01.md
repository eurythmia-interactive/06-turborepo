# **Chapter 1: Multi-App Monorepo Foundation & Tooling (Turborepo + pnpm)**

**Goal:** Establish a fast, secure, and reproducible polyglot-ready monorepo environment utilizing the latest ESM-first standards, flat configuration linter frameworks, and strict engine constraints.

---

### **Task 1.1: Workspace Scaffolding & Directory Architecture**

_Objective: Initialize the physical workspace filesystem, lock dependencies via workspace declarations, and prepare directories for applications, packages, and shared resources._

- **Subtask 1.1.1: Package Manager Initialization & Corepack Configuration**
  - Initialize Corepack to force-pin the project’s package manager to the latest stable **pnpm 11** release.
  - Configure the root `package.json` with the required `packageManager` field pinning the explicit **pnpm 11** version to prevent developers from accidentally running scripts with legacy versions or alternative manager clients.
  - Create a global `config.yaml` or update local `.npmrc` to adopt **pnpm 11** defaults, explicitly setting `blockExoticSubdeps` to true and configuring the secure, unified `allowBuilds` mapping to restrict third-party lifecycle execution out of the box.
- **Subtask 1.1.2: Define Monorepo Workspace Paths**
  - Create a `pnpm-workspace.yaml` manifest file at the root of the project.
  - Declare the workspace scope strictly to search for modules inside `apps/*` (for executable application servers) and `packages/*` (for reusable utility boundaries and internal configurations).
- **Subtask 1.1.3: Directory Tree Scaffolding**
  - Create the following physical folders at the monorepo root:
    - `apps/web`: Next.js 16+ application boundary.
    - `apps/api`: NestJS 11+ application boundary.
    - `packages/shared`: Runtime contracts, data transfer schemas, and TS declarations.
    - `packages/database`: Prisma 7 schema management, migrations, and database seed configurations.
    - `packages/config-eslint`: Shared modern lint configurations utilizing flat configuration architectures.
    - `packages/config-typescript`: Master and sub-project compiler configurations.
- **Subtask 1.1.4: Base Package Manifests Generation**
  - Generate a baseline `package.json` at the root directory of the repository with general repository details, workspace dependency definitions, and standard workflow script triggers.
  - Initialize localized, baseline `package.json` manifests in each of the app and package directories, matching the workspace workspace schema naming conventions (such as `@repo/shared`, `@repo/database`, `@repo/eslint-config`, and `@repo/tsconfig`).

---

### **Task 1.2: Turborepo Pipeline Orchestration (Turborepo 2.9+)**

_Objective: Configure the Turborepo compilation pipeline to safely orchestrate script execution with optimized parallel tasking and output caching._

- **Subtask 1.2.1: Master Turborepo Setup**
  - Install the latest **Turborepo 2.9** version as a root development dependency.
  - Create a `turbo.json` configuration file at the root of the monorepo.
- **Subtask 1.2.2: Defining Task Pipelines & Task Dependency Graphs**
  - Define core tasks under the `tasks` configuration field:
    - `build`: Configure the build process to depend on the build tasks of its internal workspace dependencies (using the caret notation `^build`). Specify build output folders (e.g., `.next/**`, `dist/**`) in the `outputs` array to allow Turborepo to cache build outputs.
    - `lint`: Set up a linting task to run in parallel across all workspaces. Enable independent caching of lint results.
    - `test`: Define the testing execution parameters. Specify that testing tasks run once dependency compile gates are clear.
    - `dev`: Define a persistent development task with watch modes enabled, explicitly turning off persistent file caching (`"cache": false`) for long-running processes.
    - `db:generate`: Establish a database generation task that outputs localized Prisma client files before compilation tasks run.
- **Subtask 1.2.3: Environment Variable & Caching Auditing**
  - Add explicit environment variable declarations inside `turbo.json`'s `globalEnv` array (e.g., standard Node/Nest configurations) to ensure that changes to critical runtime environment variables correctly invalidate the Turborepo cache.
  - Declare workspace-specific variable overrides within task structures to maintain robust cache boundaries between frontend variables and secure backend tokens.
- **Subtask 1.2.4: Local and CI/CD Remote Caching Setup**
  - Configure Turborepo command hooks inside workspace scripts to utilize safe fallback caches.
  - Establish integration points for future continuous integration workflows by mapping cache signatures and utilizing temporary file caches or linking configurations.

### **Task 1.3: Unified Monorepo Tooling (ESLint 10+, Prettier 3+)**

_Objective: Implement code style enforcement and analysis gates across the monorepo utilizing ESLint’s flat configuration format and unified formatting plugins._

- **Subtask 1.3.1: Modern ESLint Flat Configuration Design**
  - Install **ESLint 10** at the root of the monorepo as a development dependency.
  - Initialize ESLint configurations utilizing flat config files (`eslint.config.js`) instead of legacy legacy JSON or YAML file structures.
  - Create the primary configuration package inside `packages/config-eslint`. Write reusable config configurations implementing global rules for both frontend (React, Next.js) and backend (NestJS/TypeScript) apps.
- **Subtask 1.3.2: Integrating TypeScript ESLint & Flat Compat Helpers**
  - Add the latest **TypeScript ESLint** plugins to parse modern TypeScript codebases correctly under flat configs.
  - Utilize ESLint v10's built-in `includeIgnoreFile` helper from `eslint/config` to import pattern rules directly from `.gitignore` files rather than managing separate ignore files manually.
- **Subtask 1.3.3: Prettier 3 Integration**
  - Install **Prettier 3** as a root package and define standard format configurations (e.g., trailing comma rules, semicolons, and exact quote styling rules) inside a unified `.prettierrc` file at the root.
  - Install and configure ESLint Prettier integration plugins (`eslint-config-prettier`) to systematically disable style conflicts, leaving visual formatting exclusively to Prettier and logical code analysis to ESLint.
- **Subtask 1.3.4: Oxlint Pre-Flight Optimization**
  - Install the Rust-based linter tool **Oxlint** inside the monorepo to handle fast, pre-commit validation steps.
  - Integrate Oxlint commands into pre-flight development checks to provide sub-second syntax audits for developers before slower, type-aware ESLint checks run in the CI pipeline.

### **Task 1.4: Node.js Engine & Environment Controls**

_Objective: Secure the monorepo runtime by strict enforcement of Node.js engine boundaries, package manager configurations, and path-mapping tools._

- **Subtask 1.4.1: Node.js 24/26 Engine Constraint Setup**
  - Declare strict execution environment expectations within the root `package.json` package configuration under the `engines` field, setting the minimal acceptable Node.js platform version to **Node.js 24 LTS / 26 Current**.
  - Set up custom engine validation warnings inside the pnpm configuration boundaries to block standard execution if developers try to run scripts using older Node versions.
- **Subtask 1.4.2: Strict Workspace Dependency Resolution Config**
  - Establish local `.npmrc` file settings at the monorepo root to configure strict resolution.
  - Configure `save-exact=true` to lock down version drift of newly installed third-party libraries across workspaces.
  - Explicitly activate `hoist=false` or configured public hoisting limits to prevent apps from accidentally importing transitively resolved, undeclared packages.
- **Subtask 1.4.3: Global TypeScript Base Settings & Path Mapping**
  - Create the master configuration layout inside `packages/config-typescript`. Set up base configuration definitions such as `tsconfig.base.json` targeting standard Node engine configurations.
  - Configure path alias templates to make sure all packages resolve internal files systematically (using specific naming rules, like `@repo/shared/*` or `@repo/database/*`) rather than using complex relative directories.
- **Subtask 1.4.4: Root Git Ignore and Safety Checkpoints**
  - Generate a comprehensive global `.gitignore` file that prevents developers from committing local environment values, private lock states, coverage metadata, diagnostic build outputs, and the SQLite backing indexes introduced in **pnpm 11**.
