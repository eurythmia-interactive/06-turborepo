# **Chapter 3: Shared Validation Contracts (Zod)**

**Goal:** Establish a standalone schema-validation contract library within `packages/shared` using **Zod 4**, exporting TypeScript source directly (consistent with the monorepo's source-export convention) to provide compile-time and runtime validation of requests across both NestJS (backend) and Next.js (frontend).

**Approach:** Source-export model — consumers compile `@repo/shared` TypeScript directly via their own `tsc`/bundler. No separate build step (tsup/esbuild) is needed at this scale. This matches how `packages/database` already operates.

---

### **Task 3.1: Shared Library Workspace Configuration**

*Objective: Configure `packages/shared` to declare Zod as a runtime dependency and maintain its source-export package manifest.*

- **Subtask 3.1.1: Package Manifest Update**
    - `packages/shared/package.json` declares `@repo/shared` with `main`, `types`, and `exports` all pointing to TypeScript source (`./src/index.ts`).
    - `zod@4.4.3` added as a runtime dependency (schemas are validated at runtime, not just for types).
    - `build` script remains `tsc -p tsconfig.json --noEmit` (type-check only, no JS output).
- **Subtask 3.1.2: Compiler Configuration**
    - `packages/shared/tsconfig.json` extends `@repo/config-typescript/tsconfig.lib.json`, inheriting strict type checking (`strict`, `noImplicitAny`, `strictNullChecks`, `strictFunctionTypes` via `tsconfig.base.json`).
    - Module resolution is `Bundler` (matching the lib preset), compatible with both Next.js and NestJS consumers.

---

### **Task 3.2: Core Authentication Schemas (Zod 4)**

*Objective: Design validation schemas using Zod 4's top-level format helpers to secure the standard authentication boundaries of registration, login, and user profile updates.*

- **Subtask 3.2.1: User Registration Contract** (`src/auth/register.schema.ts`)
    - `email`: uses Zod 4's top-level `z.email()` helper.
    - `name`: optional trimmed string (1–100 chars), aligned with Prisma `User.name`.
    - `password`: min 8, max 128 chars, must contain uppercase + lowercase + digit + special character (regex checks).
    - `confirmPassword`: cross-field match via `.refine()` to catch client mismatch before backend logic.
    - Exports `registerSchema` and inferred `RegisterInput` type.
- **Subtask 3.2.2: Session Login Contract** (`src/auth/login.schema.ts`)
    - `email`: uses `z.email()` with a non-revealing error message.
    - `password`: non-empty string with a generic error message to prevent user enumeration.
    - Exports `loginSchema` and inferred `LoginInput` type.
- **Subtask 3.2.3: User Profile Update Contract** (`src/auth/profile.schema.ts`)
    - Base schema defines `name` (string, 1–100 chars) and `image` (URL, nullable) — matching Prisma `User` model's user-editable fields.
    - `.partial()` applied so the frontend can submit partial updates (e.g., only `name` or only `image`).
    - Exports `profileUpdateSchema` and inferred `ProfileUpdateInput` type.
- **Subtask 3.2.4: Unified Contract Export Boundary** (`src/index.ts`)
    - All auth schemas and inferred types re-exported from the primary entry point alongside existing `HEALTH_CHECK`/`ApiResponse` exports.
    - Consumers import from `@repo/shared` (e.g., `import { registerSchema, type RegisterInput } from '@repo/shared'`).

---

### **Task 3.3: Static Type Extraction**

*Objective: Programmatically generate TypeScript interfaces from Zod schemas to ensure compile-time synchronization and eliminate manual interface duplication.*

- **Subtask 3.3.1: Explicit Type Inference**
    - Each schema file exports a `z.infer<typeof schema>` type alias (`RegisterInput`, `LoginInput`, `ProfileUpdateInput`).
    - Types are co-located with their schemas for discoverability.
- **Subtask 3.3.2: Cross-Application Interface Exports**
    - All inferred types are re-exported from `src/index.ts`, ensuring IDE auto-completion works in both Next.js pages and NestJS controllers without additional imports.
- **Subtask 3.3.3: Form Resolver Compatibility**
    - Inferred types are plain TypeScript interfaces, fully compatible with `@hookform/resolvers/zod` and React Hook Form's `zodResolver`. No type casting required on client-side form submissions.

---

### **Task 3.4: Consumer Integration**

*Objective: Ensure dependent applications can consume shared schemas without additional build steps.*

- **Subtask 3.4.1: Workspace Dependency Linking**
    - Both `apps/web` and `apps/api` already declare `"@repo/shared": "workspace:*"` as a dependency. pnpm's workspace protocol creates symlinks automatically — no additional install step needed.
- **Subtask 3.4.2: Path Resolution**
    - `apps/web` resolves `@repo/shared` via `tsconfig.next.json` `paths` mapping to `../../packages/shared/src/index.ts`.
    - `apps/api` resolves `@repo/shared` via `package.json` `exports` field with NodeNext module resolution.
    - Both consumers compile shared's TypeScript source directly through their own `tsc`/bundler pipelines.
- **Subtask 3.4.3: Development Workflow**
    - Schema changes in `packages/shared/src/` are immediately visible to consumers on next type-check or dev-server hot reload — no rebuild step required.
    - Turbo's `build` task already depends on `^build`, ensuring shared is type-checked before consumers build.
