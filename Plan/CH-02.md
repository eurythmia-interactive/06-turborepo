# **Chapter 2: Relational Database & Prisma 7 Layer**

**Goal:** Configure a local database service and establish a schema-first ORM layer using Prisma’s TypeScript-based configuration format, implementing an identity model designed for SaaS applications.

---

### **Task 2.1: Containerized Local Database Setup**

_Objective: Establish a persistent, reproducible PostgreSQL runtime environment using containerization to isolate the development database from host configurations._

- **Subtask 2.1.1: Database Service Definition**
  - Create a `docker-compose.yml` configuration file in the monorepo root.
  - Define a PostgreSQL container service utilizing the lightweight `postgres:16-alpine` base image to maintain stability and low host overhead.
  - Map the host port `5432` directly to the container's standard port `5432`, binding the connection strictly to the host's loopback interface (`127.0.0.1`) to prevent external network access during local testing.
- **Subtask 2.1.2: Persistent Volume Configuration**
  - Declare a named volume (e.g., `pg_data`) inside the global `volumes` configuration block in the compose file to ensure database files survive container restarts, rebuilds, and system reboots.
  - Mount the named volume path to the container’s internal PostgreSQL data directory path (`/var/lib/postgresql/data`).
- **Subtask 2.1.3: Environment Security Mapping**
  - Specify container parameters (`POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`) using reference bindings to variables defined in the root-level `.env` file instead of hardcoding credentials in the YAML file.
  - Update the root `.env.example` file to include placeholders for database passwords, usernames, and connection targets.
- **Subtask 2.1.4: Database Readiness Health Checks**
  - Implement a container health check block using the `pg_isready` command with configured interval times, timeout thresholds, and retry limits.
  - This health verification blocks dependent services (like the backend API) from executing start scripts until the database database port is ready to accept incoming connection pools.

### **Task 2.2: Modern Prisma Configuration Architecture (Prisma 7+)**

_Objective: Implement Prisma 7's configuration files, separating environment connection variables from database model files, and utilizing TypeScript-native generation engines._

- **Subtask 2.2.1: Workspace Installation**
  - Navigate to the `packages/database` directory to configure the database package boundary.
  - Add **Prisma 7** as a development dependency and the TypeScript-native library `@prisma/client` as a runtime dependency.
- **Subtask 2.2.2: Configuration Setup (`prisma.config.ts`)**
  - Create the core Prisma configuration file `prisma.config.ts` in the root of the database package.
  - Import the `defineConfig` helper and the `env` loader from the `prisma/config` module.
  - Configure the config object to declare explicit relative paths to the model files (`prisma/schema.prisma`), database migrations directories, and datasource configurations, parsing the connection URL securely via `env("DATABASE_URL")`.
- **Subtask 2.2.3: Schema Declarations & Rust-Free Generation**
  - Create the primary `schema.prisma` file under the `packages/database/prisma` directory.
  - Inside the schema, configure the `generator client` block utilizing the new, Rust-free `prisma-client` provider (which replaces the legacy `prisma-client-js` engine) to reduce resource usage and speed up query processing.
  - Define the `output` path property inside the generator block, pointing Prisma to compile its generated client files to a local build directory (e.g., `packages/database/dist/client`) so they can be securely linked into backend and frontend applications.
  - Configure the `datasource` block to use the `"postgresql"` provider, omitting direct `url` credentials from the file since environment variables are loaded and resolved programmatically in `prisma.config.ts`.

### **Task 2.3: Core Identity Database Schema**

_Objective: Map the relational database models to represent SaaS-ready user accounts, multi-provider credentials, and token-based access control structures._

- **Subtask 2.3.1: User Entity Definition**
  - Declare the `User` model in `schema.prisma` as the central actor in the system.
  - Configure fields for primary identification (using auto-generated UUID values as database keys), audit properties (`createdAt` and `updatedAt` tracking timestamps), basic profile strings, and account status fields.
- **Subtask 2.3.2: Multi-Provider Authentication Mapping**
  - Create a secondary `AuthenticationProvider` model linked to the `User` table through a standard one-to-many relationship.
  - Add tracking fields for identity provider categories (such as `LOCAL`, `GITHUB`, or `GOOGLE` mapped to a custom schema Enum), provider-specific identifiers, and securely encrypted credential hashes.
- **Subtask 2.3.3: Roles & Permission Definitions**
  - Define a Role enumeration schema (such as `ADMIN`, `MEMBER`, and `GUEST`) to control authorization limits across application routes.
  - Design clean relationship connections mapping users to roles, allowing for a structured Role-Based Access Control (RBAC) foundation.
- **Subtask 2.3.4: Stateless Session Token Lifecycles**
  - Design a `RefreshToken` model mapped directly to the `User` table to secure the stateless JWT rotation pipeline.
  - Add fields for storing the cryptographically secure token string, creation dates, family tracking keys (for token reuse detection), and strict expiration timestamps.

### **Task 2.4: Database Client Module & Connection Pooling**

_Objective: Expose the generated Prisma client to external application workspaces as a configured, pool-optimized singleton to avoid leaking connection pools during runtime._

- **Subtask 2.4.1: Global Client Singleton Implementation**
  - Create the main entry file `packages/database/src/index.ts`.
  - Implement a global client caching pattern to ensure that hot-reloads during local development do not spawn duplicate Prisma connection instances, preventing pool exhaustion on the local database.
- **Subtask 2.4.2: Connection Pool Configurations**
  - Configure optimal connection limits and pooling timeouts (e.g., adjusting maximum connection pool thresholds to match concurrent API limits) to guard against system starvation under load.
  - Expose direct-url configuration references for migrations while using optimized transaction-pooled connection strings for application runtime processes.
- **Subtask 2.4.3: Diagnostic Logging Subscriptions**
  - Configure client initialization settings to subscribe to internal log channels, capturing raw SQL queries, warnings, and system errors.
  - Implement conditional log outputting to print database queries to console outputs during local debug sessions while silencing verbose queries in production.

### **Task 2.5: Automated Migrations and Seeding Procedures**

_Objective: Orchestrate schema state progression using Prisma's migration CLI and implement a repeatable, non-destructive initial seeding routine._

- **Subtask 2.5.1: Database Migration Commands**
  - Expose workspace script shortcuts in `packages/database/package.json` to trigger `prisma migrate dev` for development environments and `prisma migrate deploy` for production pipelines.
  - Ensure migrations are recorded inside the localized migrations folder within the database package.
- **Subtask 2.5.2: Safe Seeding Architecture**
  - Create a TypeScript seed script at `packages/database/prisma/seed.ts`.
  - Structure the database seeding logic strictly around Upsert routines to allow the script to be run multiple times against the same database without creating duplicate keys or causing index constraint failures.
  - Programmatically seed baseline system definitions, including required application roles, access permissions, and a default administrative profile for local testing.
- **Subtask 2.5.3: Prisma 7 Seed Command Configuration**
  - Register the seed script configuration inside the `migrations` block of `prisma.config.ts`, specifying the runner command (such as `tsx prisma/seed.ts` to execute TypeScript directly without compiling).
  - Establish instructions for running seeding tasks using Prisma 7's updated CLI guidelines, requiring developers to run `prisma db seed` explicitly instead of relying on automated migration triggers.
