# **Chapter 6: Next.js 16 Presentation Layer & Dashboard Client**

**Goal:** Build a high-performance web dashboard utilizing **Next.js 16** and **React 19.2**, implementing **Tailwind CSS v4** zero-config style sheets, accessible **shadcn/ui** layout primitives, and type-safe forms tied to shared workspace contracts.

---

### **Task 6.1: Next.js Client Ingestion & Base Configuration (Next.js 16 + React 19)**

*Objective: Initialize the Next.js client app, configure compilation pipelines using Turbopack, and refactor routing layouts to comply with Next.js 16's asynchronous request guidelines.*

- **Subtask 6.1.1: Next.js 16 Workspace Bootstrap**
    - Initialize the `apps/web` workspace using **Next.js 16.2** (or the latest stable 16.x line) and **React 19.2**.
    - Configure the local workspace `package.json` to enable Turbopack by default for development environments (`next dev --turbo`), taking advantage of Turbopack’s file system caching to reduce startup overhead.
    - Configure the React Compiler support within `next.config.js` to enable automatic component memoization.
- **Subtask 6.1.2: Asynchronous Request Adaptations**
    - Configure page layouts and route parameters to align with Next.js 16's strict asynchronous request handling rules.
    - Ensure that components parsing dynamic route parameters (`params`), query arguments (`searchParams`), request context headers (`headers()`), or client cookies (`cookies()`) access these properties exclusively via asynchronous await resolutions to prevent compilation warnings or runtime errors.
- **Subtask 6.1.3: Next.js 16 Cache Components Strategy**
    - Implement Next.js 16’s custom **Cache Components** architecture, using the new `"use cache"` directive to control static and dynamic layout boundaries.
    - Configure partial pre-rendering (PPR) models inside dashboard views, wrapping dynamic, user-specific widgets in Suspense boundaries while caching static template segments.
- **Subtask 6.1.4: React 19 View Transitions & Layouts**
    - Configure global navigation actions to leverage React 19’s native View Transitions support, creating smooth transitions during page navigations.
    - Define core viewport settings, metadata parameters, and SEO tags inside the primary App Router `layout.tsx` file using Next.js 16’s typed Metadata API objects.

### **Task 6.2: Accessible Design Tokens & Component Libraries (Tailwind CSS v4 + shadcn/ui)**

*Objective: Implement a fully responsive, utility-first design layer using Tailwind CSS v4's CSS-only configurations and accessible shadcn/ui component interfaces.*

- **Subtask 6.2.1: Zero-Config Tailwind CSS v4 Setup**
    - Install **Tailwind CSS v4** as a peer dependency inside `apps/web`.
    - Configure the global CSS entry stylesheet, discarding legacy `tailwind.config.js` and PostCSS configuration files.
    - Establish design tokens and theme extensions inside the stylesheet using Tailwind v4’s native `@theme` CSS variable declarations, setting custom parameters for SaaS branding colors, border radii, typography rules, and animations.
- **Subtask 6.2.2: shadcn/ui Component Library Ingestion**
    - Initialize **shadcn/ui** inside the client workspace utilizing the unified CLI (`npx shadcn@latest init`).
    - Direct the CLI to compile UI components to `components/ui`, using the CSS variables defined in Tailwind CSS v4's theme layer.
- **Subtask 6.2.3: Layout Primitives Architecture**
    - Scaffold core responsive layout components: Sidebar Navigation, Page Headers, Breadcrumb Trails, and Toast Alerts using Radix-backed primitive layers.
    - Implement automatic, system-aware light/dark mode triggers using class-based Tailwind triggers.

---

### **Task 6.3: Client-Side Form Validation Integration**

*Objective: Build user input validation pathways using React Hook Form, mapped to shared Zod 4 schemas and utilizing React 19's modern form handling states.*

- **Subtask 6.3.1: Form Configuration with Shared Contracts**
    - Install **React Hook Form** and the official Zod-to-form resolver package in `apps/web`.
    - Import inferred type structures (such as `RegisterInput` and `LoginInput`) and validation schemas directly from `@repo/shared` to enforce compile-time type safety on client-side inputs.
- **Subtask 6.3.2: React 19 Form Actions & Action States**
    - Integrate client-side forms with React 19's form action protocols.
    - Utilize React 19’s native **`useActionState`** hook to process form submissions, capturing pending execution states, operation results, and validation messages without manual state setters.
- **Subtask 6.3.3: Accessible Validation Errors Feedback**
    - Configure the UI to catch validation failures from Zod resolvers, rendering inline alerts beneath individual input fields.
    - Utilize accessible labels to ensure validation errors are readable by assistive technologies and screen readers.
- **Subtask 6.3.4: Client-Side Optimistic UI Updates**
    - Incorporate React 19’s native **`useOptimistic`** hook on real-time dashboard inputs (such as profile settings, toggle states, and item removals).
    - Ensure the client UI updates immediately while waiting for network confirmation calls, gracefully rolling back layout states if backend endpoints return errors.

### **Task 6.4: Unified Fetch Interface & Interceptors**

*Objective: Construct an HTTP request client equipped with automated bearer token injection, cross-origin credential passing, and token refresh mechanisms.*

- **Subtask 6.4.1: API Client Wrapper Construction**
    - Develop a custom fetch client configuration at `apps/web/src/lib/api-client.ts`.
    - Set up default behaviors, including automated request timeout safeguards, JSON payload stringification, explicit content type headers, and credential transmission rules (`credentials: 'include'`).
- **Subtask 6.4.2: Automated Authorization Token Injection**
    - Build client request interceptors to scan local state memory or persistent browser cookie stores for access tokens.
    - Ensure request headers automatically append Bearer credentials on outbound requests targeting protected backend API routes.
- **Subtask 6.4.3: Automated Token Renewal Interceptors**
    - Create response interceptors to catch expired credential errors (such as `401 Unauthorized` or token expiry signals).
    - When expired token errors are received, pause current pending network calls, invoke the `/api/v1/auth/refresh` endpoint in the background to rotate access cookies, and automatically retry the original request with the fresh token context.
- **Subtask 6.4.4: Concurrent Request Queue Management**
    - Build a request queue helper inside the interceptor.
    - Ensure that multiple concurrent requests hitting the client during token rotation do not trigger duplicate refresh calls; instead, queue the requests and resolve them once the single token-rotation request completes.

### **Task 6.5: Middleware Guard Architecture (proxy.ts & Next.js 16 Cookie handling)**

*Objective: Establish secure layout guards and route rewrite maps using Next.js 16's network boundary rules to protect private dashboard routes.*

- **Subtask 6.5.1: Next.js 16 proxy.ts Configuration**
    - Create a **`proxy.ts`** network boundary file at the root of `apps/web`, implementing Next.js 16's updated layout controls to replace legacy App Router middleware.
    - Configure proxy rules to coordinate routing parameters, enforce security headers, and rewrite request parameters for cross-origin backend API routes.
- **Subtask 6.5.2: Server-Side Cookie Parser Logic**
    - Write a server-side session validator utility inside `apps/web/src/lib/session.ts`.
    - Configure secure, read-only extraction of access and refresh state parameters from Next.js 16’s async-resolved cookie contexts.
- **Subtask 6.5.3: App Router Guard Implementation**
    - Create layout-level security checkpoints within protected App Router layout files (such as `apps/web/src/app/dashboard/layout.tsx`).
    - Configure checks to verify that the active user possesses a valid session. If session checks fail, block rendering of the dashboard layout and redirect the browser to the public `/login` route.
- **Subtask 6.5.4: Shared Context Providers**
    - Build a global tenant and auth user context provider at `apps/web/src/providers/auth-provider.tsx`.
    - Expose session details to nested Client Components while minimizing layout flash and content shifts during initial boot sequences.
