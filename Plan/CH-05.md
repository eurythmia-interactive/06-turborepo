# **Chapter 5: Stateless Authentication & Security Layers**

**Goal:** Implement a secure, cookie-based, stateless JWT (JSON Web Token) authentication architecture using **NestJS 11** and **Express v5**, utilizing secure cookie scopes, asymmetric signing algorithms (RS256), and automated refresh token rotation.

---

### **Task 5.1: Token Signing & Encryption Schemes**

*Objective: Establish secure token generation and signature processes using Node's high-performance native cryptographic tools and asymmetric encryption standards.*

- **Subtask 5.1.1: Cryptographic Key Pair Configuration**
    - Configure the system to sign tokens using the asymmetric **RS256** (RSA Signature with SHA-256) algorithm to decouple signing capabilities from verification.
    - Set up a production strategy where private keys (used exclusively by the NestJS authentication module to sign tokens) and public keys (used across other services to verify signatures) are read from securely mounted volumes or runtime environment configurations.
    - Configure a fallback system utilizing highly secure Symmetric HMAC-SHA256 secrets for local development environments to ensure rapid developer onboarding without manually generating certificate pairs.
- **Subtask 5.1.2: NestJS JWT Module Configuration**
    - Install the modern **`@nestjs/jwt`** package as a core dependency inside `apps/api`.
    - Register the token service using a dynamic async provider config inside `src/auth/auth.module.ts`.
    - Configure two distinct token issuance rules:
        - **Access Token:** Short-lived expiration (e.g., exactly 15 minutes) containing essential user details, role scopes, and the tenant context.
        - **Refresh Token:** Longer-lived expiration (e.g., exactly 7 days) containing a unique session identifier and cryptographic family key to prevent session hijacking.
- **Subtask 5.1.3: Token Payload Architecture**
    - Design clean, non-revealing payload interfaces. Ensure access token payloads include standard JWT registered claims (such as `sub` for user identity, `iss` for issuer, `aud` for audience, `iat` for issued-at time, and `exp` for expiration) alongside custom claims (such as roles, tenant permissions, and access levels).
    - Omit highly sensitive context parameters (such as password hashes, direct database indexes, or plain email addresses) from payloads to prevent exposure if a client-side token is inspected.

### **Task 5.2: Secure Cookie Transmission Mechanics**

*Objective: Secure the transmission of refresh tokens using Express v5 middleware and strict cookie properties to mitigate cross-site scripting (XSS) and cross-site request forgery (CSRF) vulnerabilities.*

- **Subtask 5.2.1: Express v5 Cookie Parser Integration**
    - Install the standard **`cookie-parser`** package and its corresponding TypeScript typings inside `apps/api`.
    - Register `cookie-parser` as a global middleware layer inside the Express v5 bootstrap context in `src/main.ts` before route initialization, enabling automatic parsing of HTTP headers into accessible request cookies.
- **Subtask 5.2.2: Strict Refresh Cookie Properties**
    - Design a dedicated token transmission utility inside `src/auth/utilities/cookie.utility.ts`.
    - Configure outbound refresh token cookie writes with maximum safety parameters:
        - `httpOnly: true`: Blocks client-side JavaScript access via browser scripting, neutralizing common cross-site scripting (XSS) threat vectors.
        - `secure: true`: Restricts cookie transmission exclusively to encrypted TLS/HTTPS connections (forcing this behavior in production while allowing HTTP overrides in local development contexts).
        - `sameSite: 'lax'`: Enforces cross-site request forgery (CSRF) protection for cross-origin navigations.
        - `path: '/api/v1/auth/refresh'`: Restricts browser cookie transmission exclusively to the token renewal endpoint, preventing other API controllers from processing unnecessary cookie payloads.
- **Subtask 5.2.3: Cross-Origin Resource Sharing (CORS) Configuration**
    - Enable credential-backed CORS rules in NestJS's bootstrap layer using programmatic definitions.
    - Enforce a strict policy where `credentials: true` is paired with an explicit list of origins resolved from environment parameters, explicitly rejecting wildcard origins () which would otherwise block secure cookie transfers.

### **Task 5.3: Stateless Route Guard Implementations**

*Objective: Create highly performant authentication guards in NestJS to verify access tokens on incoming requests, appending session context to execution pathways.*

- **Subtask 5.3.1: Custom Access Token Guard Construction**
    - Create a reusable authentication guard at `src/auth/guards/jwt-auth.guard.ts` implementing the NestJS `CanActivate` pipeline.
    - Extract incoming tokens from the standard Authorization header payload, parsing standard Bearer formats.
- **Subtask 5.3.2: Token Integrity Verification**
    - Inject the token validation service to verify signatures and analyze cryptographic parameters on incoming access tokens.
    - Implement robust validation checks, raising immediate `UnauthorizedException` errors if tokens are found to be expired, malformed, or signed by an untrusted source.
- **Subtask 5.3.3: Execution Context Enrichment**
    - Once validated, unpack the payload context and bind the extracted user object directly to the incoming execution request.
    - This ensures that downstream controllers and interceptors can seamlessly read user metadata (such as the user's role or tenant ID) from the request context.
- **Subtask 5.3.4: Route Decorators Design**
    - Develop a custom decorator (e.g., `@CurrentUser()`) at `src/common/decorators/current-user.decorator.ts` to simplify controller method parameter binding.
    - Develop an `@Public()` decorator to bypass JWT guard checks on open public endpoints (like health probes and landing pages).

### **Task 5.4: Role-Based Authorization Guards (RBAC)**

*Objective: Build access-control guards to evaluate user roles against route-specific requirements, blocking unauthorized actions.*

- **Subtask 5.4.1: Roles Metadata Decorators**
    - Create a custom decorator at `src/auth/decorators/roles.decorator.ts` utilizing NestJS's metadata binding tools.
    - Allow development controllers to annotate routes with acceptable role lists (e.g., specifying that only users with `ADMIN` roles can delete resources).
- **Subtask 5.4.2: Hierarchical RBAC Guard Design**
    - Develop an authorization guard at `src/auth/guards/roles.guard.ts` that runs immediately after the authentication guard.
    - Configure the guard to read the defined metadata expectations of both the method and its parent controller class, matching these parameters against the role claims mapped in the user context.
- **Subtask 5.4.3: Action-Based Permission Scopes**
    - Design modular permission validation schemas to support fine-grained action checks (e.g., evaluating permissions like `user:write` or `billing:read` rather than broad, monolithic roles like admin/user).
    - Ensure any evaluation failure triggers a clear `ForbiddenException` response, notifying the client of insufficient privilege.

### **Task 5.5: Session Revocation & Token Rotation**

*Objective: Build a secure refresh token rotation system (RTR) to invalidate session family lines upon detecting token reuse, neutralizing stolen session cookies.*

- **Subtask 5.5.1: Token Rotation Exchange Pipeline**
    - Create the token renewal endpoint `/api/v1/auth/refresh`.
    - Configure the route controller to parse the HTTP-Only cookie, extract the refresh token, and query the database to find the matching token record.
- **Subtask 5.5.2: Cryptographic Token Reuse Detection**
    - When a user requests a new token set, implement a transactional swap: invalidate the used refresh token and issue a new access/refresh pair, writing the updated credentials back to the database.
    - **Breach Mitigation:** If a user submits a refresh token that has already been used, flag this as a potential session hijack. Instantly revoke the entire token family (invalidating all active refresh tokens associated with that user profile) to force all active sessions to log out and re-authenticate.
- **Subtask 5.5.3: Session Revocation (Logout Controller)**
    - Build a logout endpoint at `/api/v1/auth/logout`.
    - Ensure the route controller deletes the active refresh token record from the database.
    - Instruct the controller to write an empty cookie back to the browser with an immediate expiration date (`expires: new Date(0)`) to ensure the browser clears the refresh token cookie.
